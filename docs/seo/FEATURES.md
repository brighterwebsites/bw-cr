# SEO module — feature spec

**Status:** draft · **Parent:** [`README.md`](README.md) · **Reference:** [`REFERENCE-SEO-OS.md`](REFERENCE-SEO-OS.md)

Priority tags: **MUST** · **SHOULD** · **LATER** · **OUT**

---

## Product goal

Give Brighter Websites a single CRM view of managed sites: **performance at a glance**, **actionable SEO opportunities across all assets**, **page-level context** (including WP meta), and a path to **human or agent tasks** — without duplicating SEO OS’s VPS/SQLite/Hermes split.

---

## F1 — Performance dashboard

### MUST

| ID | Feature | Acceptance |
|---|---|---|
| F1.1 | **28-day GSC snapshot cards** per managed asset | One card per `assets` row where `asset_type = managed_website` and latest snapshot exists |
| F1.2 | Metrics shown | Clicks, impressions, CTR, avg position — each with delta vs prior period |
| F1.3 | Data source | Latest row in `metrics_snapshots` for that `asset_id` (already in v1 schema) |
| F1.4 | Cross-asset view | All managed sites on one screen; no per-asset-only route required |
| F1.5 | Empty state | Clear when GSC not connected or no snapshot yet |

### SHOULD

| ID | Feature | Notes |
|---|---|---|
| F1.6 | Period label | Display `period_label` (e.g. “Last 28 days”) |
| F1.7 | GA4 on same card | Conversions from snapshot; engagement/session duration when populated |
| F1.8 | Link to asset | Click card → Assets detail for that site |

### LATER

- Trend sparklines / historical chart (multiple snapshots)
- Domain rank from DataForSEO on card (column exists on snapshots)

### OUT

- Rebuilding SEO OS Command Center KPI wall wholesale (approvals count, system health, etc.)

## F2 — SEO opportunities pipeline

### MUST

| ID | Feature | Acceptance |
|---|---|---|
| F2.1 | **Cross-site opportunities table** | All open opportunities; filter by asset (like SEO OS client bar) |
| F2.2 | Columns | Asset, page path, problem, priority, impressions, clicks, CTR, position, recommended workflow, status |
| F2.3 | Priority filter | All / high / medium / low |
| F2.4 | Multiple opps per page | Same URL can have many rows (e.g. low CTR + page speed) |
| F2.5 | Problem text | Human-readable summary generated by **rules**, not ad-hoc LLM per row |
| F2.6 | Promote to task | Action creates `tasks` row linked to opportunity |
| F2.7 | Status lifecycle | open → task_created → complete / dismissed |

### SHOULD

| ID | Feature | Notes |
|---|---|---|
| F2.8 | Opportunity types | Enum: `low_ctr`, `page_speed`, `striking_distance`, `content_refresh`, `serp_gap`, `indexation`, … |
| F2.9 | Evidence drawer | `evidence_json` shows thresholds hit, expected CTR band, lost clicks estimate |
| F2.10 | Sort default | Impressions descending (matches SEO OS) |
| F2.11 | Command Center preview | Top N opportunities on a future SEO home |

### LATER

- Auto-assign recommended workflow from type → playbook name
- CTR test tracking as sub-state of opportunity
- Bulk dismiss / snooze

### OUT

- Storing full GSC query × page export in Postgres
- LLM-generated problem strings on every scan row

## Explicitly OUT (this product)

| Item | Reason |
|---|---|
| SEO OS VPS SQLite + sync bridge | Supabase is source of truth |
| Review management (Zernio) | Different workflow; not in stated priorities |
| Telegram onboarding | BW-CRM uses customers/projects/assets |
| Content pipeline kanban | Later; opportunities list is enough first |
| Public multi-tenant accounts | Solo agency; Supabase auth sufficient |

## Non-functional requirements

| Area | Requirement |
|---|---|
| **Token efficiency** | Opportunity scan = scripts + rules; no LLM in the hot path |
| **Data volume** | Cap stored pages to sitemap/GSC-known URLs; expire dismissed opps |
| **Idempotency** | Re-running scan upserts by natural keys, does not duplicate opps |
| **Audit** | WP writes logged (PARKED `wp_command_log`) |
| **Secrets** | Never in RLS-visible columns; Supabase Vault / Edge secrets only |

## Agreed decisions (2026-07-22)

| # | Decision | Choice |
|---|---|---|
| 1 | Navigation | Top-level **SEO** with sub-routes: **Performance**, **Opportunities**, **Pages** — shared asset filter bar on each |
| 2 | SCOS WP meta keys | Vanessa to document exact `post_meta` keys; blocked for Phase C WP sync until then |
| 3 | GSC / GA4 auth | **Per client (per asset)** — separate OAuth grant and vault secret per `managed_website` asset |
| 3b | Admin setup UX | **Duplicate setup from…** copies non-secret `config` from another asset; **does not** copy tokens. New asset must complete its own Connect OAuth. Prevent same token/property binding on two assets. |
| 4 | Pages vs Opportunities | **Two screens** (unchanged) |
| 5 | Page metrics history | **Current-only** first |
| 6 | Opportunity scan runtime | Supabase Edge Function + pg_cron |
